01 Scoring Framework

How features are ranked

Each feature is scored on four dimensions (1–10 each). The composite score drives rank order, but strategic fit acts as a modifier a low-impact feature with very high strategic value can move up, and vice versa. All scores are PM judgment calls made explicit so they can be challenged.

User impact how meaningfully does this improve the experience for a real Pixel user?
Strategic fit does this strengthen Google's on-device AI moat vs Apple and Samsung?
Feasibility how achievable is this given Nano's current capabilities and Pixel hardware?
North Star
Make Gemini Nano the reason people choose Pixel over iPhone
Not "another AI feature" a measurably better daily-use experience that Apple and Samsung cannot replicate on their hardware. Every item on this backlog is evaluated against this framing: does it create a story a Pixel buyer tells their friend, or is it infrastructure that no one notices?
02 Stack-Ranked Backlog

8 features, ranked

01
Multi-language Recorder summarization
Critical gap vs Samsung On-device only LoRA adapter work
Gemini Nano Recorder summarization is English-first. Samsung Transcript Assist supports 17 language variants for transcription with full summary support. For a 400M+ device company competing globally, English-only summarization is a market exclusion problem, not a nice-to-have. LoRA fine-tuning (the same approach used to build the current English summarizer) can be applied per language the infrastructure exists. Start with top 5 Pixel markets by non-English usage: Japanese, Korean, Portuguese (Brazil), German, Hindi.

02
Expand Speaker Labels beyond US English
On-device Current gap: US English only LoRA / multilingual model work
Recorder's Speaker Labels feature (shipped 2022, powered by Turn-to-Diarize) already identifies and generically labels distinct speakers on-device but it is restricted to US English only. This is a significant gap given that Pixel's fastest-growing markets (Japan, Korea, Brazil, India) are non-English. A meeting recorded in Portuguese or Japanese today produces a flat, speaker-blind transcript. Expanding Turn-to-Diarize to match the top 5 non-English Pixel markets would directly complement the multilingual summarization work in item #1, and close a competitive gap vs Samsung Transcript Assist which handles speaker labeling in multiple languages. The core diarization infrastructure exists on-device; this is a language-model and data-collection expansion problem, not a net-new capability build.

03
Nano-powered offline assistant (Gemini without internet)
Fully offline Strongest Pixel moat High complexity
Gemini Live requires an internet connection. A lightweight on-device Gemini experience for common assistant tasks (set timer, summarize notification, draft reply, quick calculation, unit conversion) that works with zero connectivity would be the single most compelling Pixel differentiator. Apple's Siri has some offline capability but not with the LLM layer. Samsung has no equivalent. The "ask Gemini offline" story writes itself. Scoped to a defined task set (not open-ended chat), this is technically achievable on Nano v3 + Tensor G5.

04
Nano summarization for longer documents (10,000+ words)
On-device Context window expansion
The current ML Kit GenAI Summarization API caps at 3,000 English words. This covers news articles but not long PDFs, research papers, full meeting transcripts, or book chapters. Gemini Nano v3 on Pixel 10 has expanded token support the Recorder team used this to handle longer transcripts. Expanding the public Summarization API to ~15,000 words would open the feature to students, researchers, and professionals. Incremental engineering lift; significant expansion of use cases. Competitive differentiator vs Apple Notes (no word cap documented but no public benchmark data).

05
Expand on-device Voice Translate from Pixel 10 to Pixel 9 and older
On-device on Pixel 10 Pixel 9 and older: cloud only Hardware-constrained
Voice Translate on Pixel 10 is already on-device and offline-capable a genuine privacy and latency win. The problem is it launched exclusively on Pixel 10 and later, leaving Pixel 9, 9 Pro, 8, and 8 Pro (still the majority of the installed base) on a cloud-dependent translation path. Expanding on-device Voice Translate to Pixel 9 series (Tensor G4) requires assessing whether the model fits within G4's on-device budget a model efficiency and compression problem, not a new capability problem. Even a degraded-quality offline fallback that handles the 10 most common language pairs on Pixel 9 would be a meaningful expansion. Samsung's Live Translate has worked on-device with downloaded language packs across multiple generations of Galaxy hardware.

Nano on Pixel Watch 4 for on-device health insights
Cross-device On-device Smart Replies already shipped
Smart Replies on Pixel Watch 4 are already powered by Gemini Nano (via LiteRT-LM). The natural next step is health insight summarization taking sleep, activity, and heart rate data and generating a personalized daily insight on-device. Samsung does this with Now Brief and Energy Score. Apple Watch does it with basic trend descriptions. An on-device health LLM on Pixel Watch would be a meaningful wearable differentiator, and the data is already local. Privacy angle is strong: health data never leaves the device.

08
Nano-powered on-device photo description for accessibility
Accessibility On-device + private ML Kit Image Description exists
The ML Kit GenAI Image Description API already exists and is available on Pixel 9+. The missing layer is first-party integration into Photos, Camera, and screen reader accessibility flows. For visually impaired users, on-device image description that works offline and never transmits photos to a server is a meaningful quality-of-life improvement. Lower priority because the API exists this is a product integration decision, not a new capability. But the accessibility impact is disproportionately high for the users it serves, and Google has historically been strong on accessibility.

03 Explicit Tradeoffs

What this ranking sacrifices

What this prioritization optimizes for
  • Global user reach multi-language features ranked high because Pixel's growth markets are non-English
  • Competitive differentiation vs Apple features where Apple is clearly behind (offline assistant, diarization) are boosted
  • On-device purity cloud-dependent features deprioritized even when impact is high
  • Infrastructure leverage features that build on existing Nano capabilities ranked above net-new ML work
What this prioritization deprioritizes
  • Developer ecosystem growth no ML Kit API improvements ranked in top 3; developer needs treated as lower urgency than consumer features
  • Enterprise segment no enterprise-specific features (e.g., on-device DLP, admin policy compliance) in the backlog
  • Cross-app AI Gemini Live app integrations (Calendar, Maps, etc.) already shipping; not duplicated here
  • Short-term revenue none of these features are directly monetizable; all are retention/acquisition drivers
04 Open Decisions

What this backlog doesn't resolve

Should Nano features require Pixel 9+ (Nano v3) or support Pixel 6+ (Nano v1)?
Recommendation: tier features. Multilingual summarization and longer context window require Nano v3 (Pixel 9+). Speaker diarization likely requires Nano v3 + Tensor G5. Magic Compose and Smart Replies should expand to Pixel 6+ where feasible. Fragmentation is a real risk a two-tier Gemini experience could confuse the Pixel value proposition.
Is offline assistant (#3) a Nano feature or a Gemini app feature?
Open this has organizational implications. If it's a Gemini app feature, it needs the Gemini team's buy-in and risks feature bloat. If it's a Pixel OS feature (like Recorder), it's a tighter product story but requires cross-team alignment. Recommendation: pilot as a Pixel OS capability first to establish the on-device baseline, then evaluate folding into Gemini app.
How do we measure success for multi-language summarization launch?
North Star: Recorder usage in top-5 non-English Pixel markets (Japan, Korea, Brazil, Germany, India) 30% increase in summarization feature activation rate at 90 days post-launch. Guardrail: summary quality score (human eval) must be ≥85% of English baseline for each language before GA. Leading indicator: language pack download rate in target markets within first 2 weeks.
