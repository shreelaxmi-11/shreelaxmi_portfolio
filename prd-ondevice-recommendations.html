01 Market Context

Where Netflix and Prime Video stand today

Both Netflix and Amazon Prime Video have made significant investments in AI-driven content recommendation but as of 2026, both platforms rely entirely on cloud-side inference. Recommendations are computed on servers and served to the client. There is no on-device model running personalization logic locally.

This creates a specific, underserved problem for a growing segment of users: those in low-connectivity environments, privacy-sensitive contexts, and emerging markets where mobile data is expensive. It also limits the latency ceiling for real-time contextual adaptation.

Core finding: Neither Netflix nor Amazon Prime Video currently runs AI recommendation inference on-device. Both are cloud-first architectures. This PRD treats on-device recommendations as a forward-looking product opportunity, not a description of existing capability.

Netflix

Current AI recommendation architecture

  • FM-Intent foundation model: predicts user intent and next watch simultaneously within a session
  • Ensemble methods: collaborative filtering, content-based filtering, and deep learning combined
  • Reinforcement learning for long-term satisfaction signals (not just click optimization)
  • Recommendations drive 80%+ of all viewership; estimated $1B/year in retention value
  • Smart Downloads / Downloads for You: AI pre-fetches content to device on Wi-Fi but recommendations are computed cloud-side before download
  • Real-time re-ranking based on device type, time of day, and session signals all server-side
  • 325M+ subscribers globally as of end of 2025 (Netflix stopped reporting quarterly numbers from Q1 2025); strong in emerging markets (India mobile-only plan)
No on-device inference Smart Downloads exists RL for long-term value
Amazon Prime Video

Current AI recommendation architecture

  • AI Topics (Dec 2024): Bedrock-powered content discovery system grouping titles into personalized topic clusters
  • X-Ray Recaps (Nov 2024): AI text summaries personalized to exact viewing position; reported by industry sources as among the most-used Fire TV features post-launch
  • Video Recaps (Nov 2025): theatrical-quality AI video summaries with narration, dialogue, music beta for US Prime Originals
  • All inference via Amazon Bedrock (cloud) and custom SageMaker models no on-device AI
  • X-Ray: cast, trivia, music overlays during playback (legacy; cloud-fed metadata)
  • Fire TV ecosystem advantage: tighter hardware control than Netflix across devices
  • Alexa integration enables voice-driven discovery cloud-side NLU
Bedrock/SageMaker infra No on-device inference Fire TV hardware leverage
02 Problem Statement

The connectivity gap in content discovery

Cloud-first recommendation architectures work well for high-connectivity users. They break down or degrade significantly for three distinct user segments that represent a large and growing share of streaming subscribers globally.

User segment Scale Current experience Unmet need
Commuters & travelers Estimated 40–60% of mobile sessions Downloaded content plays; no fresh recommendations without Wi-Fi or data Personalized suggestions from already-downloaded library, ranked by current context
Emerging market mobile users India, Southeast Asia, Africa fastest-growing subscriber segments Limited data plans mean users avoid loading the home screen; high data cost of fetching recommendations Zero-data-cost recommendations from on-device model; no cloud call needed
Privacy-conscious users Growing segment across EU, US post-GDPR Every viewing session and context signal is transmitted to cloud for inference Local inference that never transmits behavioral data externally
Users with slow/intermittent connectivity Rural areas; low-end devices on congested networks Home screen loads slowly; recommendation latency visible to user; cold start failures Instant, locally-rendered recommendations with zero round-trip latency

Opportunity framing: On-device recommendations do not replace cloud inference they extend it. The goal is a fallback-and-augment model: cloud recommendations when available; a lightweight on-device model that holds the experience together when they're not, and personalizes further using local signals unavailable to the server (battery level, storage state, ambient noise, time of day patterns).

Why now?

Three infrastructure shifts make on-device recommendation inference viable in 2026 in a way it wasn't at scale in 2021: (1) On-device ML runtimes have matured TFLite, Core ML, and ONNX Runtime can now serve small recommendation models at <10ms latency on mid-range devices. (2) LoRA fine-tuning enables compact, task-specific adapters on top of a shared base model the same pattern Google used to ship Gemini Nano for Recorder. (3) Federated learning allows model improvement from on-device behavior without centralizing sensitive data.

03 Proposed Solution

On-device recommendation layer: LocalLens

LocalLens is a lightweight on-device recommendation module that runs alongside the existing cloud recommendation stack. It is not a replacement it is a contextual enrichment and offline fallback layer. It maintains a compressed user preference model on-device, updated periodically via federated sync, and produces real-time recommendations using only local signals when cloud is unavailable.

Netflix · LocalLens

Feature scope

  • Offline home screen: When network unavailable, surface recommendations from downloaded library ranked by a local model not a static list
  • Downloads for You 2.0: Replace cloud-computed pre-fetch queue with a hybrid model that factors in local signals (time of day, battery, storage, location type) not available server-side
  • Context-adaptive re-ranking: Adjust home row ordering in real-time based on current session context (e.g., 11pm → re-rank toward shorter episodes)
  • Privacy mode: Opt-in local-only mode where behavioral signals are never transmitted; model stays on device
Prime Video · LocalLens

Feature scope

  • On-device X-Ray Topics: Run AI Topics discovery model locally for offline browsing; no Bedrock call required when browsing downloaded content
  • Fire TV Nano: Embed a compressed recommendation model in Fire TV OS first-party hardware advantage enables deeper OS integration than Netflix has available
  • Alexa offline suggestions: Lightweight NLU + recommendation fusion for common voice queries ("what should I watch tonight?") answered without internet
  • Contextual download queue: Pre-fetch decisions made on-device using local context signals; reduce reliance on server-side scheduling

Technical architecture

The on-device model is a compressed two-tower retrieval model (user tower + item tower) distilled from the existing cloud recommendation model. User tower runs on-device and produces a preference embedding updated after each session. Item tower is pre-computed for the downloaded library and stored locally. At inference, dot-product similarity produces a ranked list in under 5ms on a mid-range device.

ComponentApproachRationale
Model format ONNX Runtime / TFLite quantized INT8 Cross-platform; 4–8x size reduction vs FP32 with minimal accuracy loss
Model size target <50MB on-device footprint Acceptable for mid-range devices; below user-visible storage concern threshold
Update cadence Federated sync weekly over Wi-Fi Keeps model fresh without continuous data transmission; privacy-preserving
Fallback behavior Recency + genre affinity heuristics if model unavailable Graceful degradation; never shows empty state
Privacy guarantee Behavioral signals processed locally; only model gradients (not raw data) transmitted in federated update GDPR Article 25 (data protection by design); competitive trust differentiator
04 User Stories

Jobs to be done

Meera · Commuter · Mumbai
"As a daily train commuter with a patchy 4G signal, I want to open Netflix offline and see recommendations from what I've downloaded not a blank screen."
So that I can start watching immediately without waiting for the app to load recommendations from the server.

Acceptance criteria

  • Offline home screen renders <500ms with ≥5 ranked recommendations
  • Recommendations are personalized, not a static recency list
  • No network call made; zero data usage
James · Late-night viewer · Chicago
"As someone who watches at 11pm after work, I want the app to surface shorter content without me having to filter it should know what I need right now."
So that I don't have to manually scroll past 2-hour films when I only have 30 minutes.

Acceptance criteria

  • Local model ingests time-of-day as a contextual signal
  • Episode/runtime re-ranking applied within existing rows, not a new UI element
  • User study shows ≥15% reduction in time-to-play at night sessions
Priya · Privacy user · Berlin
"As a user who has opted out of personalization data sharing, I still want relevant recommendations I just don't want my viewing behavior leaving my device."
So that I get a personalized experience without trading privacy for convenience.

Acceptance criteria

  • Privacy mode: all inference runs locally, zero telemetry sent
  • Model updates via federated learning only (no raw data leaves device)
  • Privacy mode clearly indicated in UI with auditable explanation
Carlos · International traveler · São Paulo → NY
"As a frequent flyer, I want Prime Video to automatically queue up content for my 10-hour flight based on my mood and how much storage I have left."
So that I don't have to manually manage downloads before every trip.

Acceptance criteria

  • On-device model reads storage state and available Wi-Fi window
  • Download queue optimized for total runtime vs storage budget
  • Suggestions reflect recent viewing, not stale cloud preferences

06 Risks & Mitigations

What could go wrong

Dimension
Severity
Risk
Mitigation
Recommendation staleness
High
On-device model may drift from user's evolving preferences if federated updates are infrequent or device stays offline long-term
Staleness indicator after 7 days without sync; fallback to recency + genre heuristics; aggressive Wi-Fi sync scheduling
Model accuracy gap
High
Compressed on-device model will underperform the cloud ensemble if gap is too large, users notice lower relevance and trust erodes
A/B test gate: on-device model must reach ≥90% of cloud CTR before broad rollout; use online distillation to close the gap iteratively
Device fragmentation
Medium
INT8 quantized models behave differently across SoCs; inference quality varies on MediaTek vs Snapdragon vs Apple Silicon
Test on device matrix of top 20 devices by MAU; maintain SoC-specific model variants for top 5 chips; graceful fallback on unknown hardware
Storage backlash
Medium
Users notice 50MB+ model on device; negative reviews about "Netflix using my storage"
Model bundled with app update incrementally; surfaced as a setting with clear explanation; downloadable on-demand, not forced
Federated learning privacy edge cases
Low
Gradient inversion attacks could theoretically reconstruct user behavior from federated updates
Differential privacy noise added to gradients before transmission; third-party privacy audit before launch; bug bounty program
07 Phased Roadmap

From zero to on-device inference

Phase 1
Foundation
Months 1–4
  • Audit current offline experience; baseline metrics on offline session starts and time-to-play
  • Train compressed two-tower model distilled from production cloud model
  • INT8 quantization + benchmarking on top 20 devices
  • Federated learning infrastructure setup (gradient aggregation, differential privacy)
  • Internal dogfooding with 500-user panel
Exit criteriaOn-device model CTR within 10% of cloud CTR on panel; inference latency <10ms p99
Phase 2
Offline-first launch
Months 5–9
  • Ship offline home screen recommendations to 10% of Android MAU (A/B test)
  • Downloads for You v2: on-device download queue optimization with local signals
  • Context-adaptive re-ranking (time-of-day, battery state) in A/B test
  • Privacy mode opt-in (beta, invite-only)
  • Federated sync weekly; monitor gradient quality and model drift
Exit criteria+15% offline session start rate vs control; no significant NPS regression
Phase 3
Scale & differentiation
Months 10–18
  • Full Android + iOS rollout to all MAU
  • Privacy mode GA with clear UI and user-facing audit log
  • Emerging market optimization: ultra-compressed model variant (<15MB) for low-end devices
  • For Prime Video: Fire TV OS integration (OS-level model, shared across apps)
  • Publish external privacy whitepaper; third-party audit
Exit criteria+20% offline session starts; Privacy mode at 5% MAU; model footprint <50MB on all tiers

Open decisions

Should on-device and cloud recommendations blend or strictly fall back?
Recommendation: Hybrid blend when both available; on-device fallback when cloud unavailable
Blending allows local signals (time, battery, storage) to influence cloud results even when connected higher ceiling than pure fallback
Federated learning vs periodic model download?
Recommendation: Federated learning for user tower; central download for item tower
User preferences are private and benefit from federated privacy; item embeddings are not personal and can be updated centrally without privacy risk
Should Privacy mode be opt-in or opt-out in GDPR regions?
Open legal review required
GDPR Article 25 (privacy by design) may require opt-out default in EU; US and other regions likely opt-in. Decision pending legal team input
